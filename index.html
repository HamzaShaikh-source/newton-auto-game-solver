BETA Development Paused

When Simple Ideas Become Deep Learning

This project started as a quick experiment — "how hard can auto-solving block games be?" Turns out, it's a rabbit hole that touches BFS pathfinding, DOM parsing, React fiber internals, Blockly's API model, and maze-solving heuristics. Six unique games, each with its own block set, terrain rules, and edge cases.

Why is development paused?

The developer stepped back to focus on personal growth — deepening foundational CS knowledge, exploring systems-level programming, and building stronger engineering habits. This project was always a learning vehicle, and sometimes the best way to move forward is to pause, reflect, and level up the fundamentals first.

What makes this interesting

React Fiber Internals — Walking the fiber tree for Blockly workspace injection
6 Diverse Games — Unique block sets, terrain types, and goal mechanics
Maze-Solving Heuristics — Wall-follower with cycle detection and conditional blocks

Games

Six Newton School block-coding games, each with different mechanics, block sets, and challenge curves.

Levels 1-10 Duck Duck Go!

Guide a duck to collect a ball across a grid. Early levels use only forward/repeat blocks. Later levels introduce turns, conditionals (if/else), and tighter block budgets.

Goal: ball.png
Blocks: forward, turn, repeat, if/else
Terrain: rock (wall), bush (path), glass (hazard)
Levels 1-7 Mouse wants Laddoo

Navigate a mouse to a laddoo (sweet). Unique floor/tile terrain system where floor tiles are walls and tile variants are paths. Uses fixed_repeat blocks with if-do conditionals.

Goal: laddoo.png
Blocks: forward, turn, fixed_repeat, if-do
Terrain: floor (wall), tile (path)
Levels 1-10 Turtle Tracer

Steer a turtle through obstacle courses. Distinct terrain system using colored markers. Emphasizes turn optimization and path sequencing with limited block budgets.

Goal: burger.png
Blocks: forward, turn, repeat
Terrain: floor markers, colored tiles
Levels 1-8 Boat to the Shore

Navigate a boat carrying a passenger to shore. Features water tiles, boat-specific terrain, and the passenger goal sprite. Introduces repeat_until blocks.

Goal: passenger.png / shore.png
Blocks: forward, turn, repeat_until
Terrain: water, boat landing, shore
Levels 1-6 A Friend In Need

Similar structure to other navigation games but with its own terrain palette and block constraints. Uses the burger goal and introduces tighter block budgets at higher levels.

Goal: burger.png
Blocks: forward, turn, repeat, if/else
Terrain: mixed ground/stone tiles
Levels 1-6 Hungry Duck

Same underlying engine as Duck Duck Go but with a hamburger goal. Shares block types and terrain mechanics. Useful for cross-validating the solver's terrain detection against different asset filenames.

Goal: burger.png
Blocks: forward, turn, repeat, if/else
Terrain: rock (wall), bush (path), glass (hazard)
Cross-game note: Terrain mapping is game-specific. For example, floor.png is a wall in mouse-laddoo but a path in duck-ball. The solver handles this via game-specific override blocks in readGameState().

Dev Log

Chronological record of difficulties encountered, how they were solved, and what's still broken.

2026-05-28 OPEN

Wall-follower repeat count overshoots the goal

Problem: The maze solver's wall-follower algorithm sets a repeat count of 10 iterations, but the level only needs 8 forward moves. The extra 2 iterations cause the character to walk past the goal and fail.

Root cause: The simulation counts turns as separate steps, but the generated code folds conditional turns into forward iterations. The repeat count needs to be the number of forward moves, not simulation steps.

Fix status: Not yet fixed. Need to compute forward-only step count in simulateWallFollower() and pass it to the repeat count.

2026-05-28 OPEN

Player direction detection fallback may be wrong

Problem: detectInitialDirection() returns direction 2 (down) as fallback for mouse-laddoo, when the mouse sprite actually faces right (direction 1).

Attempted fix: Added CSS class rotation detection (rotateimg270 etc.) which correctly detects right-facing, but the fallback path when no sprite icon is found still defaults to the wrong value.

Fix status: Works when sprite icon has rotation CSS class. Needs robust fallback when icon element structure differs.

2026-05-27 FIXED

DOM grid cell ordering inconsistent across games

Problem: The BFS pathfinder returned paths that didn't match the visual grid because <img> elements in some games were ordered top-to-bottom while others were left-to-right.

Fix: Used CSS Grid's computed grid-template-columns to detect column count, then arrange cells row-by-row. Fall back to bounding box positions if CSS grid info is unavailable.

2026-05-26 FIXED

Block injection cleared workspace but blocks didn't render

Problem: workspace.clear() removed blocks but workspace.newBlock() created invisible blocks. The React-managed Blockly workspace needed additional fiber tree interaction to refresh.

Fix: Found workspace via React fiber tree walk, used workspace.newBlock() + .initSvg() + .render() + workspace.render() sequence. Connected blocks programmatically before calling top-level render.

2026-05-25 FIXED

Extension not injecting solver on page load

Problem: Content script injected solver.js via a <script> tag, but the browser CSP blocked execution. The solver functions were not available when the popup tried to call them.

Fix: Listed solver.js as a web_accessible_resource in manifest.json and inserted it as a <script src="chrome-extension://..."> tag. Added __ngs_solver_injected flag to prevent double injection.

2026-05-24 FIXED

Terrain filenames URL-encoded differently per game

Problem: Some games URL-encode spaces as %20 in image src attributes while others use raw filenames. The terrain map lookup failed for encoded names.

Fix: Applied decodeURIComponent() before the terrain lookup. Lowercased filenames for case-insensitive matching.

Known issues: Wall-follower repeat count, direction detection fallback, and conditional block generation for complex mazes (levels with if/else and multiple branches).

Code Breakdown

Detailed walkthrough of every file in the project.

manifest.json

Chrome Extension Manifest V3 configuration.

{
  "permissions": ["storage", "activeTab"],
  "host_permissions": [
    "https://learn.newtonschool.co/*",
    "https://my.newtonschool.co/*"
  ],
  "content_scripts": [{
    "matches": ["https://learn.newtonschool.co/block-games/*",
                "https://my.newtonschool.co/playground/game/*"],
    "js": ["content.js"],
    "run_at": "document_end"
  }],
  "web_accessible_resources": [{
    "resources": ["solver.js"],
    "matches": ["https://learn.newtonschool.co/*",
                "https://my.newtonschool.co/*"]
  }]
}

Defines permissions, content script injection, and exposes solver.js as a web-accessible resource so the isolated-world content script can inject it into the page's main world.

content.js

Isolated-world content script that bridges the extension popup and the main-world solver.

// Listens for messages from popup
// Injects solver.js into the main world
// Relays solver results back to popup
// Handles: status check, solve request, result retrieval

The script creates a <script> tag pointing to the web-accessible solver.js and appends it to the page body. It listens for __ngs_solver_result postMessage events from the solver and stores the result for the popup. Responds to three message actions: status (is solver loaded?), solve (trigger auto-solve), getResult (retrieve stored result).

solver.js (47 KB)

The core solver engine — self-contained IIFE that runs in the page's main world.

Key functions:

readGameState() — Parses the DOM grid, detects terrain from image filenames, locates player and goal sprites by bounding box.
findPath() — BFS pathfinder that computes shortest path from player to goal, avoiding walls and hazards. Handles goal-on-wall edge case.
buildBlockSequence() — Converts path directions into optimized block sequences. Uses pattern-finding to compress repeated moves into repeat blocks.
simulateWallFollower() — Right-hand-rule maze solver with cycle detection. Generates conditional programs (if path not ahead → turn right → forward).
detectInitialDirection() — Determines player facing by checking CSS transforms, rotation classes, and game-specific fallbacks.
injectBlocks() — Walks the React fiber tree of the Blockly workspace, clears existing blocks, creates and connects new blocks programmatically.
popup.html + popup.js

Extension popup UI with auto-solve trigger, status display, and grid preview.

popup.html defines a dark-themed popup with status indicator, solve button, result area, and map preview grid.
popup.js communicates with the content script via chrome.tabs.sendMessage. On page load, it queries solver status. On solve click, it triggers auto-solve and displays the result including path length, injection success, and a color-coded grid preview.
